# Phase 2: Create an API Contract

Before writing code, define what frontend will call.

For dashboards, I recommend:

```http
GET /api/v1/dashboard
```

Single API.

Response:

```json
{
  "stats": {},
  "weekly_activity": [],
  "weekly_goal": {},
  "monthly_consistency": [],
  "milestones": [],
  "enrolled_courses": [],
  "recently_completed": []
}
```

---

# Why One API?

Without it:

```text
GET /stats
GET /courses
GET /milestones
GET /goal
GET /activity
```

Frontend makes:

```text
5-6 network requests
```

Instead:

```text
1 request
```

Dashboard loads faster.

Most modern dashboards use an aggregate endpoint.

# Phase 5: Create Response Schemas

In FastAPI:

```python
from pydantic import BaseModel

class Stats(BaseModel):
    enrolled_courses: int
    completed_courses: int
    learning_hours: float
    streak_days: int
```

---

Dashboard response:

```python
class DashboardResponse(BaseModel):
    stats: Stats
    weekly_activity: list
    weekly_goal: dict
    monthly_consistency: list
    milestones: list
    enrolled_courses: list
    recently_completed: list
```

# Phase 6: Build With Mock Data First

Do NOT wait for database completion.

Create:

```python
@router.get("/dashboard")
def get_dashboard():
    return {
        "stats": {
            "enrolled_courses": 12,
            "completed_courses": 4,
            "learning_hours": 84.5,
            "streak_days": 7
        }
    }
```

Frontend team can start immediately.

---

# Phase 7: Introduce Service Layer

Bad:

```python
@router.get("/dashboard")
def get_dashboard():
    # database logic here
```

Good:

```python
@router.get("/dashboard")
def get_dashboard():
    return dashboard_service.get_dashboard()
```

---

Service:

```python
def get_dashboard():
    stats = repo.get_stats()
    courses = repo.get_courses()

    return {
        "stats": stats,
        "courses": courses
    }
```

# Phase 8: Repository Layer

Repository only talks to DB.

```python
class DashboardRepository:

    def get_stats(self, user_id):
        pass

    def get_courses(self, user_id):
        pass

    def get_weekly_activity(self, user_id):
        pass
```

---

Flow becomes:

```text

   ↓

   ↓

   ↓

   ↓

```

# Phase 9: Think About Authentication Early

Dashboard is user-specific.

Never expose:

```http
GET /dashboard
```

without knowing the user.

Instead:

```http
GET /dashboard
Authorization: Bearer <JWT>
```

Flow:

```text
JWT

 ↓

Extract User ID

 ↓

Fetch Dashboard Data

 ↓

Return User Dashboard
```

# Phase 10: Questions You Must Ask Before Development

If no workflow exists, create one.

### Product Questions

1. How is streak calculated?
2. What makes a course "completed"?
3. Can users have multiple goals?
4. How many upcoming milestones should be shown?
5. Should completed milestones be hidden?

---

### Frontend Questions

1. Single API or multiple APIs?
2. Should data refresh automatically?
3. Pagination for courses?
4. Any sorting requirements?

---

### Database Questions

1. PostgreSQL or MySQL?
2. Existing tables?
3. Existing user authentication?

---

A good GitHub issue workflow for this task would be:

```text
1. Analyze UI
2. Extract data requirements
3. Design API contract
4. Design DB schema
5. Create FastAPI project structure
6. Implement mock endpoint
7. Connect DB
8. Add JWT auth
9. Add tests
10. Create PR
```

This approach prevents you from jumping straight into coding and later discovering that the frontend expected a completely different API structure.
